BLOCKING ISSUE #1: Must NOT Have #9 Violated — SSO/arkcli used as MAIN scheme
──────────────────────────────────────────────────────────────────────────────

Plan requirement (line 100):
  "不使用 Cookie/SSO 鉴权作为主方案（仅 arkcli fallback）"
  Translation: "Do not use Cookie/SSO auth as the main scheme (arkcli is
  only a fallback)"

Evidence of violation:
  File: src-tauri/src/volcano.rs:57-69
  ```
  fn query_volcano_usage() -> Result<VolcanoUsage, String> {
      let output = std::process::Command::new("arkcli")
          .args(["usage", "plan"])
          .output()
          ...
  ```

  arkcli uses SSO authentication and is the ONLY data source for volcano
  usage. The V4 signing module (src-tauri/src/v4_sign.rs) exists but is
  NOT called by volcano.rs. arkcli is the primary scheme, not a fallback.

  Note: This stems from a plan-internal contradiction. Task 0 spike
  confirmed GetPersonalPlan does not return usage data, and Task 5
  explicitly instructs: "如果 GetPersonalPlan 不返回用量数据：实现调用
  `arkcli usage plan` 命令". However, the Must NOT Have guardrail is
  still violated regardless of the contingency path.

Severity: GUARDRAIL VIOLATION (plan-internal contradiction)

──────────────────────────────────────────────────────────────────────────────
BLOCKING ISSUE #2: Tauri v2 Invoke Argument Naming Mismatch (CRITICAL RUNTIME BUG)
──────────────────────────────────────────────────────────────────────────────

Tauri v2 official documentation confirms:
  "Arguments are passed from JavaScript as a JSON object with camelCase
  keys by default, but you can use #[tauri::command(rename_all =
  "snake_case")] to match snake_case arguments."
  Source: https://v2.tauri.app/develop/calling-rust

The Rust commands do NOT use `rename_all = "snake_case"` (grep confirmed
zero matches). Therefore, Tauri v2 expects camelCase keys from JS.

But the frontend uses snake_case keys — a "recent fix" that is INCORRECT.

Mismatched arguments (6 of 7 are wrong):

1. save_credentials (App.tsx:259-263):
   Frontend passes:  volcano_access_key, volcano_secret_key, deepseek_api_key
   Tauri expects:    volcanoAccessKey, volcanoSecretKey, deepseekApiKey
   Result: ALL 3 credential fields will be empty/missing at runtime

2. save_settings (App.tsx:264-267):
   Frontend passes:  refresh_interval, opacity
   Tauri expects:    refreshInterval, opacity
   Result: refresh_interval will be missing (opacity is correct — single word)

3. get_deepseek_balance (useUsageData.ts:66-68, 75-77):
   Frontend passes:  api_key
   Tauri expects:    apiKey
   Result: api_key will be empty/missing → DeepSeek API call fails

Impact on plan requirements:
  - Must Have #3 (DeepSeek 余额): BROKEN at runtime — get_deepseek_balance
    receives empty api_key, cannot fetch balance
  - Must Have #9 (API 密钥加密存储): BROKEN at runtime — save_credentials
    receives empty strings, saves empty credentials to keyring
  - DoD #4 (DeepSeek 余额数据正确显示): UNVERIFIABLE — balance fetch fails
  - DoD #7 (设置界面可配置 API 密钥并保存): UNVERIFIABLE — save fails

Note: cargo check and npm run build PASS because TypeScript does not
type-check Tauri invoke argument names against Rust signatures. The bug
only manifests at runtime.

The learnings.md (line 395-398) CORRECTLY documented camelCase as the
convention, but the "recent fix" reversed it to snake_case, introducing
this regression.

Fix: Revert frontend invoke args to camelCase:
  - volcano_access_key → volcanoAccessKey
  - volcano_secret_key → volcanoSecretKey
  - deepseek_api_key → deepseekApiKey
  - refresh_interval → refreshInterval
  - api_key → apiKey

Severity: CRITICAL — breaks 3 of 7 Tauri commands at runtime
